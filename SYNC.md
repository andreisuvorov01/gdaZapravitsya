# Синхронизация данных с gdebenz.ru

Два скрипта тянут данные о заправках РФ из публичного API сообщества
**gdebenz.ru** и складывают их в Postgres этого проекта (self-hosted, см. `CLAUDE.md`):

- **`scripts/sync-gdebenz.mjs`** — обход тайлов `/api/stations` по всей
  России: находит станции (координаты, бренд, адрес), `upsert`'ит их в
  таблицу `stations` и пишет одну грубую текущую запись статуса на станцию.
  Это единственный источник НОВЫХ станций — запускать первым/регулярно.
- **`scripts/sync-gdebenz-comments.mjs`** — быстрый, отдельный скрипт: НЕ
  ходит по тайлам и не трогает таблицу `stations`, а читает уже известные
  станции из БД (по колонке `gdebenz_id`, которую проставляет скрипт
  выше) и для каждой дёргает `/api/comments/<id>/recent` — богатую ленту
  последних отметок (топливо + очередь + лимит на руки + точное время каждой
  отметки, а не одна текущая сводка). См. [«Лента отметок»](#лента-отметок-по-станции-apicommentsidrecent)
  ниже.

Общий HTTP/прокси/browser-транспорт (ретраи, ротация прокси, фоллбэк
transports) вынесен в `scripts/lib/gdebenz-http.mjs`, разбор статусов/топлива —
в `scripts/lib/gdebenz-parse.mjs`; оба скрипта используют одни и те же модули.

> **Будьте уважительны к источнику.** Серверы gdebenz.ru поддерживаются на
> пожертвования. Запросы троттлятся (`THROTTLE_MS`, по умолчанию 300 мс).
> Не запускайте полный обход России чаще, чем нужно, и соблюдайте условия
> использования сервиса.

## Миграция БД

`sync-gdebenz-comments.mjs` требует колонки `stations.gdebenz_id` и
`stations.gdebenz_comments_synced_at` (обе уже в `db/schema.sql`) — примените
схему перед первым запуском:

```bash
npm run setup:db
```

Затем нужен хотя бы один прогон `sync-gdebenz.mjs`, чтобы проставить
`gdebenz_id` для существующих станций (или дождаться обычного расписания).

## Переменные окружения

Берутся из `.env` (через `scripts/load-env.mjs`). Общие для обоих скриптов
(читаются из `scripts/lib/gdebenz-http.mjs`):

| Переменная | Обязательность | Назначение |
| --- | --- | --- |
| `DATABASE_URL` | да | Строка подключения к Postgres |
| `THROTTLE_MS` | опционально | Базовая пауза, мс (по умолчанию `300`). От неё считаются более короткие browser/HTTP задержки |
| `REQUEST_TIMEOUT_MS` (или старое имя `TILE_TIMEOUT_MS`) | опционально | Общий таймаут на один запрос со всеми фоллбэками |
| `BROWSER_TIMEOUT_MS` | опционально | Таймаут browser-транспорта. По умолчанию больше HTTP, чтобы он был основным и не срывался на медленных ответах |
| `HTTP_TIMEOUT_MS` | опционально | Таймаут HTTP-fallback'а. По умолчанию короче browser, чтобы ошибки отрабатывали быстрее |
| `BROWSER_RESPONSE_DELAY_MS` | опционально | Пауза после browser-ответа |
| `HTTP_RESPONSE_DELAY_MS` | опционально | Пауза после HTTP-ответа. Обычно короче browser |
| `RETRY_DELAY_MS` | опционально | Пауза перед повторной попыткой после ошибки |
| `MAX_RETRIES` | опционально | Число попыток на запрос (по умолчанию `3`) |
| `FETCH_TRANSPORTS` | опционально | Порядок транспорта запроса. По умолчанию `browser,https4`. `https6` можно добавить вручную, если на хосте есть рабочий IPv6 |
| `PROXY_LIST` | опционально | Список прокси через запятую. Поддерживаются HTTP(S) (`"http://user:pass@host:port"`) и SOCKS4/5 (`"socks5://user:pass@host:port"`) — схема в URL определяет, какой агент используется, смешивать оба типа в одном списке можно. Прокси берутся по кругу — по одному на **попытку** запроса, так что при блокировке одного IP повтор автоматически уходит через следующий прокси. Имеет приоритет над `PROXY_URL` |
| `PROXY_URL` | опционально | Один прокси на все запросы (тот же формат, что и один элемент `PROXY_LIST`), если `PROXY_LIST` не задан |

> **SOCKS5 и browser-транспорт.** Chromium (через Playwright) не умеет
> авторизовываться на SOCKS5-прокси — это ограничение самого браузера
> (`Browser does not support socks5 proxy authentication`), не нашего кода.
> Для авторизованных SOCKS5-прокси `browser`-транспорт по умолчанию
> автоматически пропускается (без траты попытки/времени), запрос сразу уходит
> через `https4`. Если у вас именно SOCKS5-прокси с логином/паролем — это
> ожидаемое поведение, а не баг; проверить, что прокси реально работают,
> можно через `node scripts/check-proxies.mjs` (см. ниже).
>
> Если `browser`-транспорт для SOCKS5 всё-таки нужен (он лучше переживает
> TLS-фингерпринтинг/анти-бот проверки, чем голый HTTPS-клиент) — запустите
> локальный HTTP↔SOCKS5 мост отдельным процессом:
> ```bash
> npm run proxy:bridge   # держите открытым, пока идёт синк
> ```
> Мост поднимает по одному локальному НЕ-авторизованному HTTP-прокси
> (`127.0.0.1:18080`, `18081`, … — по одному на каждый прокси из
> `PROXY_LIST`, порт = `PROXY_BRIDGE_BASE_PORT` (по умолчанию `18080`) + номер
> прокси в списке) и сам логинится на реальный SOCKS5 с вашим логином/паролем.
> Затем запустите синк с `PROXY_BRIDGE=1` — `browser`-транспорт пойдёт через
> мост вместо прямого (невозможного) подключения к SOCKS5:
> ```bash
> PROXY_BRIDGE=1 npm run sync:gdebenz:comments
> ```
> Без запущенного моста `PROXY_BRIDGE=1` ничего не сломает — если мост не
> слушает нужный порт, `browser`-транспорт просто провалится и упадёт в
> `https4`-фоллбэк, как раньше.

> **Зачем прокси.** gdebenz.ru может троттлить/блокировать IP после большого
> числа запросов подряд — при полном обходе РФ с одного адреса это выглядит
> не как явная ошибка, а как тихо опустевшие тайлы (`fetchTile` после
> исчерпания ретраев просто пишет `console.warn` и возвращает `[]`). В сводке
> `sync-gdebenz.mjs` есть счётчик `tilesFailed` — если он стабильно больше
> нуля, часть станций каждый раз молча выпадает из БД, и это повод подключить
> `PROXY_LIST`.

Специфичные для `sync-gdebenz.mjs` (обход тайлов):

| Переменная | Назначение |
| --- | --- |
| `SYNC_BBOX` | Синхронизировать ТОЛЬКО этот bbox `"south,west,north,east"` (тоже тайлами). Можно вместо переменной передать тем же форматом аргументом CLI |
| `TILE_LAT`, `TILE_LON` | Размер тайла в градусах (по умолчанию `1.2`). Меньше тайл → больше запросов, но надёжнее против `bbox_too_large` |
| `SYNC_CONCURRENCY` | Число параллельных воркеров по тайлам (по умолчанию `1` на Windows, `2` на Linux) |
| `BATCH`, `SELECT_CHUNK` | Размер батча upsert/insert и чанка `select ... in (...)` |
| `DENSITY_MODE` | `full` (по умолчанию) или любое другое значение — берёт каждый 4-й тайл (для быстрой проверки покрытия) |

Специфичные для `sync-gdebenz-comments.mjs` (лента отметок):

| Переменная | Назначение |
| --- | --- |
| `COMMENTS_LIMIT` | Сколько последних отметок запрашивать на станцию у `/api/comments/<id>/recent` (по умолчанию `20`) |
| `COMMENTS_CONCURRENCY` | Число параллельных запросов внутри одной страницы станций (по умолчанию `1` на Windows, `2` на Linux). Реальный потолок частоты запросов теперь держит `COMMENTS_MIN_INTERVAL_MS`, а не это число — конкурентность в основном скрывает сетевую задержку, поднимать её без причины не нужно |
| `COMMENTS_MIN_INTERVAL_MS` | Минимальный интервал, мс, между СТАРТАМИ любых двух запросов к gdebenz.ru, независимо от конкурентности (по умолчанию `500` — не больше 2 запросов/сек на весь процесс). Главный рычаг «не упереться в rate limit» — жёстче и предсказуемее, чем настройка одной конкурентности |
| `COMMENTS_COOLDOWN_MIN` | Не запрашивать ленту станции повторно, если её уже успешно забирали в последние N минут (по умолчанию `25`). Резко сокращает число запросов на повторных прогонах — обрабатываются только станции, которые реально устарели. Держите ниже интервала расписания, иначе плановый прогон будет считать всё "свежим" и пропускать |
| `COMMENTS_BREAKER_THRESHOLD` | Сколько неудачных запросов подряд считать признаком блокировки по IP и останавливать весь прогон, вместо того чтобы продолжать долбить уже заблокированный сервер (по умолчанию `6`) |
| `PAGE_SIZE` | Сколько станций тянуть из БД и обрабатывать за одну "волну" (по умолчанию `500`) — определяет, насколько часто коммитятся промежуточные результаты |
| `SYNC_BBOX` | Ограничить станции по `lat`/`lng` в этом bbox (для точечного теста) |

> **Как это работает вместе.** Каждый прогон обрабатывает только станции,
> где `gdebenz_comments_synced_at` пуст или старше `COMMENTS_COOLDOWN_MIN`
> (курсорная пагинация по `id`, устойчивая к тому, что фильтр "сжимается" по
> ходу прогона — см. комментарий в коде). Внутри — `COMMENTS_MIN_INTERVAL_MS`
> не даёт превысить частоту запросов вне зависимости от конкурентности. Если
> `COMMENTS_BREAKER_THRESHOLD` неудач подряд — прогон останавливается сам, не
> дожидаясь, пока это сделаете вы. Успешно опрошенные станции (в т.ч. с
> пустой лентой) помечаются `gdebenz_comments_synced_at = now()`;
> провалившиеся — нет, и будут повторены на следующем прогоне.

### Флаги `--all` / `--turbo`

```bash
npm run sync:gdebenz:comments:turbo
# то же самое, что:
node scripts/sync-gdebenz-comments.mjs --all --turbo
```

- `--all` — игнорирует cooldown на этот прогон (`COMMENTS_COOLDOWN_MIN=0`):
  обходит вообще все станции с `gdebenz_id`, а не только "устаревшие".
- `--turbo` (или `--fast`) — заметно быстрее ценой более высокого риска
  повторно словить блок по IP (см. инцидент выше): `COMMENTS_MIN_INTERVAL_MS=100`
  (~10 запросов/сек вместо ~2), `COMMENTS_CONCURRENCY=5`,
  `COMMENTS_BREAKER_THRESHOLD=12` (терпимее к шуму на более высокой частоте).
  Это не проверенное на практике безопасное значение, а осознанный компромисс
  "быстрее, но не буквально то же самое, что уже один раз словило блок" —
  если он снова словит 403, брейкер остановит прогон сам, не сжигая весь
  оставшийся список станций впустую.
- Явно заданные переменные окружения (`COMMENTS_MIN_INTERVAL_MS=...` и т.п.)
  всегда имеют приоритет над обоими флагами — можно, например, взять `--all`
  для полного охвата, но подставить свой, более консервативный интервал.
- Единственный способ реально ускориться БЕЗ роста риска на один IP —
  `PROXY_LIST` с несколькими прокси: тогда нагрузка размазывается по разным
  адресам вместо того, чтобы упираться в лимит одного.

## Лента отметок по станции (`/api/comments/<id>/recent`)

Помимо тайлового API (одна текущая сводка на станцию за тайл), gdebenz.ru
отдаёт по каждой станции последние N отметок пользователей —
`GET /api/comments/<gdebenz_id>/recent?limit=20`, где `<gdebenz_id>` — тот же
id, что вернул тайловый API (числовой OSM id или `usr_...` для станций,
добавленных через сам gdebenz). Ответ — плоский JSON-массив:

```json
[{"status":"yes","detail":"92, 95, 98, 100, ДТ · Большая очередь","created_at":"2026-07-02 09:58:53","author_reliable":false}]
```

`detail` — не жёсткий формат, а список топлива плюс произвольные аннотации
через " · " (очередь словами или "≈N–M машин", лимит "Лимит N л"), парсится
`parseGdebenzDetail()` в `scripts/lib/gdebenz-parse.mjs`. `created_at` —
наивная строка без таймзоны, трактуется как UTC (`parseGdebenzTimestamp()`).

`sync-gdebenz-comments.mjs` читает станции из БД страницами по
`PAGE_SIZE`, внутри страницы дёргает эту ленту конкурентно
(`COMMENTS_CONCURRENCY`), и сразу после страницы пишет результат в БД —
не копит всё в памяти до конца прогона. Для станции, где запрос удался,
**вся** предыдущая история `client_id='gdebenz'` этой станции удаляется и
заменяется полученной лентой целиком (у отметок нет стабильного id, поэтому
полная замена на каждом прогоне проще и не плодит дублей между соседними
прогонами, чем попытка слить старое с новым по времени). Станции, где запрос
не удался (сеть/блокировка) или гдебenz ничего не вернул, просто пропускаются
на этом прогоне — их старая история остаётся как есть.

Это один запрос на станцию, а не на тайл ~1.2°×1.2° — при полном обходе РФ
это десятки/сотни тысяч запросов к gdebenz.ru за прогон, на порядок больше,
чем у `sync-gdebenz.mjs`. Обязательно настройте `PROXY_LIST`/
`COMMENTS_CONCURRENCY` разумно для регулярного полного обхода, чтобы не
создавать источнику чрезмерную нагрузку, и используйте `SYNC_BBOX` для
точечных тестов.

> **Если ловите `403 Forbidden` (нередактируемая HTML-страница nginx) даже на
> `GET /` без параметров** — это не троттлинг конкретного запроса, а блок
> IP целиком на уровне nginx/хостинга gdebenz.ru. Судя по всему, срабатывает
> от резкого всплеска запросов (что и делает этот скрипт при высокой
> `COMMENTS_CONCURRENCY` без прокси). В этой ситуации:
> 1. Прекратите слать запросы с этого IP — повторные попытки во время бана
>    ничего не решают и могут выглядеть как продолжение злоупотребления.
> 2. Подождите (часы, не минуты) и проверьте вручную одним запросом
>    (`curl -I https://gdebenz.ru/`), прежде чем снова гонять скрипт.
> 3. Если блок не снимается — это может быть блокировка по диапазону
>    IP хостинг-провайдера (датацентр), а не именно вашего адреса; в этом
>    случае обычный datacenter-прокси из `PROXY_LIST` не поможет, нужен
>    residential/mobile-прокси.
> 4. Самый надёжный вариант — списаться с администрацией gdebenz.ru и
>    договориться о разрешённой частоте опроса, а не наращивать
>    конкурентность/прокси в одностороннем порядке.

## Ручной запуск (что чаще всего нужно)

Все команды требуют `.env` с `DATABASE_URL`, а также установленных
зависимостей (`npm ci` — подтянет в т.ч. `pg`).

### Через npm-скрипты

```bash
# 1. Станции + грубый статус: полный обход населённой РФ (надолго, используйте бережно)
npm run sync:gdebenz

# 2. Лента отметок по уже известным станциям (быстрее, но тоже много запросов на полной БД)
npm run sync:gdebenz:comments

# Только один регион. Bbox передаётся аргументом после "--"
# (формат "south,west,north,east"; пример — центр Москвы):
npm run sync:gdebenz -- "55.6,37.4,55.85,37.75"
npm run sync:gdebenz:comments -- "55.6,37.4,55.85,37.75"
```

### Напрямую через node (рекомендуется на сервере Beget)

```bash
cd /path/to/app

# Станции + грубый статус, полный обход РФ
node scripts/sync-gdebenz.mjs

# Лента отметок по уже известным станциям
node scripts/sync-gdebenz-comments.mjs

# Только один bbox — аргументом (работает и в bash, и в PowerShell)
node scripts/sync-gdebenz.mjs "55.6,37.4,55.85,37.75"
# либо именованным флагом
node scripts/sync-gdebenz.mjs --bbox=55.6,37.4,55.85,37.75
```

Тот же bbox можно задать переменной окружения (если так удобнее):

```bash
# bash / Linux (Beget)
SYNC_BBOX="55.6,37.4,55.85,37.75" node scripts/sync-gdebenz.mjs
```

```powershell
# PowerShell (Windows)
$env:SYNC_BBOX="55.6,37.4,55.85,37.75"; node scripts/sync-gdebenz.mjs
```

Оба скрипта идемпотентны. `sync-gdebenz.mjs`: станции `upsert` по `osm_id`,
грубые статусы помечаются `client_id='gdebenz'` и на каждом запуске
**добавляются** новыми строками со свежим `created_at` (попадают в окно
свежести 3 ч из `lib/freshness.ts`); старые отчёты не удаляются пачкой перед
вставкой — чистятся только строки старше 2 суток. Это специально сделано так,
чтобы точечный синк по узкому `?bbox=` не стирал данные по остальной России.
`sync-gdebenz-comments.mjs` заменяет историю станции целиком при каждом
успешном запросе (см. выше) — тоже безопасно для узкого `SYNC_BBOX`, т.к.
удаление скоуплено списком станций текущей страницы.

## Авто-расписание

> **Где жить расписанию.** С self-hosted Postgres (слушает только localhost/
> unix-socket на самом VPS — см. `docs/SETUP-DB.md`) синк-скрипты нужно
> запускать НА ТОМ ЖЕ сервере, что и БД/приложение — с ноутбука или любого
> другого хоста `DATABASE_URL` сетевым доступом не достанется. Расписание —
> обычный cron на VPS.

Рекомендуемая частота: `sync-gdebenz.mjs` (станции + грубый статус) — раз в
3 часа, `sync-gdebenz-comments.mjs` (лента отметок) — чаще, т.к. это более
лёгкий и оперативный источник (например, раз в 30–60 минут), но не забывайте,
что `sync-gdebenz.mjs` должен запускаться первым хотя бы один раз, чтобы
проставить `gdebenz_id` для новых станций.

### Beget / Linux (рекомендуемый прод-вариант)

```cron
# Станции + грубый статус, каждые 3 часа
0 */3 * * * cd /path/to/app && DENSITY_MODE=full SYNC_CONCURRENCY=2 FETCH_TRANSPORTS=browser,https4 node scripts/sync-gdebenz.mjs >> /var/log/benzryadom-sync.log 2>&1

# Лента отметок, каждые 30 минут. Не поднимайте COMMENTS_CONCURRENCY без
# PROXY_LIST — на дефолте (2) и то возможен блок по IP, см. предупреждение
# про "403 Forbidden" ниже.
*/30 * * * * cd /path/to/app && FETCH_TRANSPORTS=browser,https4 node scripts/sync-gdebenz-comments.mjs >> /var/log/benzryadom-sync-comments.log 2>&1
```

Замените `/path/to/app` на реальный путь к проекту. Если `node` не в `PATH`
у cron — укажите полный путь к бинарю (узнать: `which node`), например
`/usr/bin/node`. На shared-хостинге (Beget) лог удобнее класть в каталог
проекта (`>> sync.log 2>&1`), если нет доступа к `/var/log`.

#### Вариант через HTTP-роут с `CRON_SECRET`

Если на сервере нет системного cron, но есть планировщик HTTP-запросов —
роут `/api/cron/sync-gdebenz` запускает только `sync-gdebenz.mjs` (см. ниже),
`sync-gdebenz-comments.mjs` через него не вызывается — гоняйте его прямым
запуском скрипта системным cron'ом.

```cron
0 */3 * * * curl -fsS -H "Authorization: Bearer $CRON_SECRET" "https://домен/api/cron/sync-gdebenz?bbox=55.6,37.4,55.85,37.75" >> /var/log/benzryadom-sync.log 2>&1
```

> **Полный обход РФ через HTTP гонять не стоит** — он длится несколько минут и
> легко упирается в таймаут прокси/serverless-функции. Через HTTP дёргайте
> только узкий `?bbox=...`; для полного обхода используйте прямой запуск скрипта
> системным cron'ом (вариант выше).

### Windows Task Scheduler (локально, dev-машина)

Создать задачи, запускающие оба синка (пути соответствуют этой машине):

```bat
schtasks /Create /TN BenzRyadomSync /TR "\"C:\Program Files\nodejs\node.exe\" \"C:\Users\Andrey\Desktop\benzin\scripts\sync-gdebenz.mjs\"" /SC HOURLY /MO 3 /ST 00:00 /F
schtasks /Create /TN BenzRyadomSyncComments /TR "\"C:\Program Files\nodejs\node.exe\" \"C:\Users\Andrey\Desktop\benzin\scripts\sync-gdebenz-comments.mjs\"" /SC MINUTE /MO 30 /F
```

Проверить, что задача создалась:

```bat
schtasks /Query /TN BenzRyadomSync
```

Запустить вручную для проверки / удалить:

```bat
schtasks /Run /TN BenzRyadomSync
schtasks /Delete /TN BenzRyadomSync /F
```

> `schtasks` сам не умеет задавать «рабочую папку» — скрипты определяют
> корень проекта относительно своего расположения (`scripts/load-env.mjs`
> читает `../.env`), поэтому достаточно передать абсолютный путь к `.mjs`.
>
> **Важно:** эти задачи сработают, только когда машина включена. Для прода
> используйте Beget/облачный cron (см. выше).

## HTTP-триггер (опционально)

`GET|POST /api/cron/sync-gdebenz` запускает `sync-gdebenz.mjs` (станции +
грубый статус) дочерним процессом и возвращает JSON-сводку. Ключ передаётся
одним из способов:

- заголовок `Authorization: Bearer <CRON_SECRET>`;
- заголовок `x-cron-key: <CRON_SECRET>`;
- query-параметр `?key=<CRON_SECRET>`.

Без заданного `CRON_SECRET` роут отвечает `503`. Для `sync-gdebenz-comments.mjs`
отдельного HTTP-роута нет — запускайте его напрямую системным cron'ом.

```bash
# Узкий регион через HTTP (безопасно по времени)
curl -H "Authorization: Bearer $CRON_SECRET" \
  "https://домен/api/cron/sync-gdebenz?bbox=55.6,37.4,55.85,37.75"
```

> **Не гоняйте полный обход РФ через HTTP.** Полный синк длится несколько минут
> и легко упирается в таймаут прокси/serverless-функции. Для полного обхода
> запускайте скрипт напрямую (`node scripts/sync-gdebenz.mjs`) или системным
> cron'ом. Через HTTP дёргайте только узкий `?bbox=...`.

## Как настроить покрытие

- Полный набор макро-регионов задан в `MACRO_REGIONS` в начале
  `scripts/sync-gdebenz.mjs`. Добавляйте/убирайте прямоугольники под нужные
  территории. Финальная отсечка по границе РФ делается отдельным полигоном в
  `scripts/data/russia-adm0-simplified.geojson`, поэтому исходные прямоугольники
  могут немного заходить за границу, но в БД такие точки не попадут.
- Размер тайлов — через `TILE_LAT` / `TILE_LON`. Если какой-то тайл вернёт
  `bbox_too_large`, скрипт сам делит его на 4 и повторяет.
- Покрытие `sync-gdebenz-comments.mjs` определяется исключительно тем, какие
  станции уже есть в `stations` с непустым `gdebenz_id` — новых территорий
  оно не открывает, только углубляет данные по уже известным станциям.
